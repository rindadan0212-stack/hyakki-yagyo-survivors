# Part I — 思考レンズ（設計の核）

## 1. 第一原理: ゲームとは「興味深い選択の連続」である（Sid Meier）
プレイヤーが毎瞬「意味のある選択」をしているか。これが面白さの根本。
**「興味深い選択」の条件:**
- 複数の選択肢が現実的に有効である（=支配的な最適解が1つに収束していない）
- 各選択に明確な結果・トレードオフがある
- 判断に必要な情報が与えられている
- 短期の得と長期の得が必ずしも一致しない

→ **テスト**: ある機能・アップグレードについて「これは興味深い選択を生むか、それとも単なる数値増加か？」を必ず問う。後者ばかりなら単調になる。

## 2. MDAフレームワーク（Hunicke, LeBlanc, Zubek 2004）
- **Mechanics（メカニクス）**: ルール・基本アクション・データ構造。設計者が直接いじれる唯一の層。
- **Dynamics（ダイナミクス）**: メカニクスがプレイヤー入力と相互作用して生まれる実行時の挙動。
- **Aesthetics（エステティクス）**: プレイヤーに生じる感情反応。

**設計者はメカニクスからアプローチし、プレイヤーはエステティクスから体験する（逆方向）。**
→ **必ず目標エステティクスから逆算する。** 「この感情を出したい → どんなダイナミクスが要る → どのメカニクスを置く」。メカニクスを足し算で増やしてから面白さを祈るのは禁止。

## 3. 8つの楽しさ（LeBlanc）— 狙うエステティクスを名指しする
1. **Sensation 感覚** — 視覚・音・手触りそのものの快。
2. **Fantasy 空想** — 別世界・なりきり。
3. **Narrative 物語** — ドラマ・展開。
4. **Challenge 挑戦** — 障害の克服・上達。
5. **Fellowship 仲間** — 社会的相互作用（主にマルチ）。
6. **Discovery 発見** — 未知の探索・秘密・新スキル。
7. **Expression 表現** — 自己表現・ビルドや見た目の選択。
8. **Submission 服従** — 没入的な反復・「気づいたら時間が溶ける」。

→ どの機能も「8つのどれを狙うか」を明示。1機能が複数を兼ねてよいが、**狙いが無い機能は作らない。**

## 4. フロー理論（Csikszentmihalyi）
挑戦と技能が釣り合う対角の帯が**フロー**。上にはみ出すと不安、下にはみ出すと退屈。
**フロー突入の条件**: 明確な目標／即時のフィードバック／技能をわずかに上回る挑戦／集中を妨げない環境。
→ プレイヤーの技能上達に合わせ**難度を上げ続ける**。難度カーブは右肩上がりが基本。
→ 動的難易度調整（DDA）は道具として使える（プレイヤーの状態で難度を微調整）。ただし露骨だと白ける。

**重要な例外**: Souls系の「克服快感（ordeal pleasure）」のように、フロー帯の上に長く留めること自体を快にする設計もある。条件は「失敗が理不尽でなく、学習可能であること」。つまりフロー帯維持は唯一解ではなく、狙うエステティクス次第。

## 5. ゲームフィール（Swink）— 瞬間の「効き」
3本柱: **①リアルタイム制御 ②模擬空間（物理・重さ・速度・当たり） ③磨き（polish=アニメ/音/粒子/カメラ）**。
- 入力→反応の**補正サイクルが約100ms以内**でないと「もっさり」する。入力遅延・反応遅延は最優先で潰す。
- 「ジューシーさ(juice)」は③の磨きの層。シミュレーション本体を変えずに体感を激変させる。
- 入力バッファ、コヨーテタイム等の「許容」は、正しさより**気持ちよさ**を優先する正当な手段。

## 6. ループ構造（コア／セッション／メタ）
- **コアループ**: プレイヤーが何度も繰り返す主要行動の連鎖（例: 敵を倒す→経験値→強化→より強い敵）。
- **セッションループ**: 1プレイ（1ラン）の始まり〜終わりの弧。
- **メタループ**: ラン跨ぎの永続進行・アンロック。

**強制ループ（compulsion loop）**: コアループを「報酬への期待」で駆動するよう調律したもの。**変動比率スケジュール**（毎回ではなくランダムな確率で報酬）で強化される。スロット的快感はこれ。
→ **倫理的な線**: 関与は「上達と報酬の納得感」で生む。射幸性の煽りやダークパターンで時間を搾取しない。長期的な信頼＝良いクラフト。

## 7. システム的フィードバックループ（正/負）
- **正のフィードバック**: 勝っている者がさらに有利に（雪だるま）。パワーファンタジーに最適、競り合いの維持には不利。
- **負のフィードバック**: 状態を安定化（ラバーバンド、追い上げ補正）。緊張維持に有効、理不尽に感じさせるリスク。
→ どちらを使うかは**意図的に選ぶ**。ヴァンサバ系は強い正のフィードバックで「神になる」体験を作るのが本質（後述 Appendix）。

## 8. 意味のある選択と二次設計問題
**プレイヤーは放っておくと「面白さを最適化で消す」**（支配的戦略を見つけて作業化する）。
→ 支配的戦略を作らない。アップグレード・武器は**プレイの仕方が変わる質的差**を持たせる（貫通・連鎖・発射形が変わる等）。単なる+%だけにしない。
→ 選択肢は「強いけど別物」を並べる。リロール/バニッシュでプレイヤーに能動性を返すのも有効。

## 9. やって学ばせる — 起承転結（4ステップ）設計（宮本/林田、Nintendo）
1. **起(導入)**: 新メカニクスを**ペナルティのない安全な場**で提示。
2. **承(展開)**: 同じメカニクスを危険・複合状況で使わせる。
3. **転(ひねり)**: 予想外の組合せ・新要素で再文脈化し、習熟を試す。
4. **結(結論)**: 学んだことを使って締め、達成を祝福する。

→ **テキストチュートリアル原則禁止。** メカニクスは安全な状況で体験的に教える。
→ これは新規ステージ/ボス/ウェーブの**遭遇設計の単位**としても使う。

## 10. 可読性とテレグラフ（Norman: アフォーダンス/シグニファイア）
プレイヤーは常に「脅威」と「選べる手」を読めなければならない。
→ 危険には予兆（テレグラフ）。重要要素はコントラスト・サイズ・色で階層化。
→ 画面が混沌としても**手前のアクションが読める**よう、背景の減彩・プレイヤーの強調を維持（特に高密度ジャンルで死活的）。
→ 本作は実装済: charge=赤い進路ライン、slam=狙い線+満ちる赤円（詠唱中に倒せば不発）。新規の脅威にも必ず予告を。

## 11. 創発（Tynan Sylvester『Designing Games』）
記憶に残る瞬間は、作り込んだスクリプトより**単純なシステム同士の相互作用**から生まれる。
→ メカニクスは**組み合わせ可能（composable）**に作る。小さな部品が掛け算で新挙動を生む方向を優先する。
→ 本作は実装済: 隠し相乗（特定スキル併せ持ちで自動発動・recipe非表示）がこの思想の具体化。

# Part II — 制作哲学（プロセス）

- **まず面白さを見つける**: コンテンツやアートより先に**コアループのプロトタイプ**で「面白いか」を確かめる。縦切り(vertical slice)で1ループを完成度高く作り、早く失敗する(fail fast)。
- **設計者はプレイヤーではない**: 自分が面白いかではなく、テストプレイで**観察**して判断する。説明したくなったら、それは設計の欠陥のサイン。
- **コアを反復、磨きは計画的に**: juiceは「あとで余裕があれば」ではなく最初から計画に入れる。ただしコアが面白くないうちに磨かない。
- **スコープ規律**: 機能は積むより削る。広さより深さ。1つの良いループ＞10個の中途半端な機能。

# Part III — 技術アーキテクチャ哲学（設計＝反復なので、系は常に可変に保つ）

> 設計はチューニングの連続。だから**コードを触らずに値と挙動を変えられる**構造にしておくことが、良い設計を可能にする前提条件になる。
> ※ここでは「原則」を述べ、本プロジェクトでの**具体的な実体**を併記する（実装規約の詳細は `HyakkiUnity/CLAUDE.md`）。

## データ駆動設計（本作の実体 = `data.js` → `gamedata.json` → `GameDB`）
- **調整値はコードに埋めずデータに置く。** 武器・敵・アップグレード・ウェーブ・宝具・技などの数値は**単一の真実源 `js/data.js`** に集約する。
- Unity版は `data.js` を `tools/export_data.js` で `Assets/Resources/gamedata.json` に**自動エクスポート**し、`GameDB`（JsonUtility）が読む。**手書き転記しない**（転記事故＝バランスの二重管理を防ぐ）。曲線関数（spawnInterval/hpScale 等）はJSON化で落ちるので C#側（`Balance`/各Sim）に**コメント付き定数**で持つ。
- 「インスペクタで実行時調整」のUnity標準解は ScriptableObject だが、**本プロジェクトはWeb版とのパリティ維持を優先しJSON駆動**を採る。調整はWeb版/`data.js`で行い再エクスポート、または `__G`/`G.debug` のデバッグAPIで実行時に叩く。
  - ※Unity単独の新規調整対象を足すならSO化も選択肢。ただし**真実源の二重化に注意**（`data.js`が正）。

## 系の疎結合（レイヤ分離）
- 系同士を直接密結合させない。本作は **Sim（純ロジック）/ Render（Batch2D）/ UI（IMGUI）/ Audio（予定）** で層分離。**Sim は表示・入力・音を知らない**＝Web版とのパリティ検証が純関数的に書ける。
- 「被弾」「撃破」等は Sim 内のカウンタ/イベントで表現し、Render・UI・（将来）Audio が**状態を読んで**独立に反応（フラッシュ・数字・シェイク・実績・絵巻）。
- これにより機能の**追加・削除が他を壊さない**。

## 継承よりコンポジション
- 巨大な継承ツリーを作らない。エンティティ（敵/弾/死体/魂/提灯/結界/霊炎/宝玉）は**プール内のプレーンなクラス**で、挙動は move型・state（FSM）など**データとフラグの組合せ**で表現する（創発しやすい）。

## その他の基盤（本作での実体）
- **ステートマシン(FSM)**: ゲーム状態 `AppState`(Title/Setup/Run/Paused/Records…) と `Phase`(Running/LevelUp/Chest/Pact…)、敵AIの move/state。暗黙のbool地獄を避ける。
- **ロジックと表示を分離**: スコア計算・状態遷移は Sim、描画は Render。テスト対象はロジック側。
- **固定タイムステップ**: `GameApp.Update` のアキュムレータで **60Hz自前ループ**（`FixedUpdate`不使用＝hitstop/slowmo/timescaleを自前制御）。
- **オブジェクトプーリング**: 敵・弾・粒子・ダメージ数字・死体は徹底プール。**Obtain時に全フィールドReset必須**（再利用バグの温床＝再三やらかした箇所）。
- **"Make it work → make it right → make it fast"**。プロファイラで詰まりを確認してから最適化する。

# Part IV — Code運用ルール（実装前チェックリスト）

機能を実装する前に、Codeは以下を明示してから着手すること:

1. **狙うエステティクス**: 8つのうちどれを狙う機能か。
2. **ループ位置**: コア/セッション/メタのどこに効くか。
3. **興味深い選択**: これはプレイヤーの意味ある選択を生むか、それとも単なる数値/演出か。後者なら役割（フィードバック/磨き）を明確に。
4. **ゲームフィール**: 入力→反応は100ms以内か。磨きの層（フラッシュ/音/シェイク/数字）は付くか。
5. **可変性**: 調整値はデータ駆動（`data.js`）か。系は直接参照でなく層分離・状態購読で疎結合か。
6. **可読性**: 高密度・混沌でも脅威と選択肢が読めるか。

## 自己レビュー（出力に対して必ず通す）
- [ ] 支配的戦略（1つの最適解）を生んでいないか
- [ ] パワーカーブが平坦になっていないか（質的変化があるか）
- [ ] 単なる+%アップグレードばかりになっていないか
- [ ] フィードバックが薄い（無反応の操作がある）まま放置していないか
- [ ] 調整値をハードコードしていないか（`data.js`に置いたか）
- [ ] 新メカニクスを安全な場で導入しているか（いきなり危険な状況で出していないか）
- [ ] 画面が混沌時に読めなくなっていないか

# Appendix — ジャンルプロファイル: ヴァンサバ系（horde survival / bullet heaven）

このプロジェクトの狙いに合わせた当てはめ。

- **主要エステティクス**: Sensation（画面を埋める大量フィードバック）＋ Challenge（生存と密度）＋ Submission（没入反復）＋ Discovery（ビルド/進化の発見）。
- **フィードバックループ**: **強い正のフィードバック**を意図的に使う。弱い序盤 → 指数的に強化 → 終盤は画面を消し飛ばす全能感。これがジャンルの本体。雪だるまを邪魔しない。
- **興味深い選択 = ビルド構築**: レベルアップ選択・スキル抽選・相乗・覚醒が「興味深い選択」の主戦場。質的アップグレードと隠し相乗で「発見」を作る。
- **ゲームフィール=ボルテージ**: 毎秒数百キルなので、1キルの磨き（ヒットフラッシュ/ダメージ数字/撃破ポップ/XPマグネット）が全体の体感を決める。ヒットストップ・シェイクは**クリ/エリート/集約撃破/レベルアップ等の重い瞬間にゲート**する。
- **ペーシング=起承転結**: ウェーブを「息継ぎ(起)→展開(承)→スウォーム/ひねり(転)→ボスや一瞬の緩和(結)」のリズムで。包絡線は右肩上がり、内側に緩急の波。スポーン強度は経過時間で駆動するディレクタにする（ベタ湧き禁止）。
- **メタループ**: 恒久アンロック・複数使い手・鍛錬・実績で、ラン跨ぎの動機（メタのボルテージ）を作る。
- **データ駆動の要**: 武器/敵/アップグレード/進化条件/ウェーブを全て `data.js` に。バランス調整を再エクスポートで回せる状態を保つ。

**本作での実装状況（2026-06時点）**: 初期武器=使い手固定（成長は宝具/鍛錬/覚醒/数打ち）、他の得物はスキル抽選（累積・再取得でLvUP）、隠し相乗7種、提灯の滞在段階、鍛錬レベル制（使い手/得物/技）、奥義カットイン（立ち絵）、レア度ドラフト、実績、百鬼絵巻、敵の charge/slam 予告攻撃、超過撃破=爆散。詳細は Claude memory を参照。**音源は移植最終段（未着手）。**
